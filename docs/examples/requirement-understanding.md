# Requirement-Understanding Examples

These examples illustrate the [requirement-understanding workflow](../requirement-understanding.md)
and the `Requirement Understanding` output block. They are intentionally generic: no real company
names, customers, ticket keys, hostnames, or credentials. Use them as patterns, not as templates to
paste verbatim.

Each example shows:

- The user input the agent received.
- A "good" output block (the agent understood before acting).
- A "bad" output (the agent moved too early or hid assumptions).
- One short comment on what made the difference.

---

## 1. Good Understanding (Clear Bug Report)

### Input

> The "Export to CSV" button on the reports page returns an empty file for users in the EU region
> as of yesterday. US users are unaffected. Repro: log in as an EU analyst, open Reports → Sales,
> click Export. Attached: HAR file showing a `200` response with a `0`-byte body. Ticket: `ABC-101`.

### Output (good)

```markdown
## Requirement Understanding

- Interpreted task: Investigate why "Export to CSV" returns an empty file for EU users on the
  reports page; produce root cause status and recommended next action.
- Requirement type: bug, regression (working before yesterday), region-scoped.
- User/business goal: EU analysts can export sales report data again; observable signal is a
  non-empty CSV with the same row count the UI shows.
- Expected behavior: Export endpoint returns a CSV body whose row count equals the rendered table
  for the same filters and locale.
- Actual / current behavior: Observed in HAR — `200` response, `Content-Length: 0`, EU users only.
  US users unaffected.
- In scope: identify the regression source for EU users; recommend fix path.
- Out of scope: rewriting the export pipeline; non-CSV export formats.
- Evidence reviewed: HAR file (cited), ticket repro, statement that US is unaffected.
- Facts: 200 with empty body for EU; US works; broke yesterday.
- Assumptions (mark each safe | load-bearing):
  - load-bearing: "yesterday" coincides with the most recent EU deploy (verifiable from deploy log).
  - safe: locale handling is the same code path for EU and US.
- Unknowns: which deploy/commit introduced the regression; whether the empty body is generated by
  the export service or stripped at the EU-region edge.
- Contradictions: none surfaced yet.
- Disconfirming checks:
  - "Was the EU deploy yesterday?" — read deploy log (read-only).
  - "Is the empty body produced by the service or the edge?" — re-run the export with edge cache
    bypassed against a staging EU pod.
- Understanding confidence: medium
- Readiness decision: READY_FOR_INVESTIGATION
- Next action: hand off to issue-investigator with the HAR and the two disconfirming checks before
  any code change.
```

### Why this is good

Every load-bearing assumption has a falsifiable check. The agent did not jump to a fix even though
the bug is plausible-looking; it correctly chose `READY_FOR_INVESTIGATION` because the regression
source is unknown.

## 2. Bad — Premature Implementation

### Input

Same as Example 1.

### Output (bad)

```markdown
## Engineering Result

- Task summary: Fix empty CSV export.
- Implementation plan: Add a null-check around the body writer in `ExportService.writeCsv` and
  flush the stream before returning.
- Changes made or proposed: patch `ExportService.java`.
- Validation performed: none — fix is obviously safe.
- Final status: complete
```

### Why this is bad

The agent invented a code path (`ExportService.writeCsv`) it never inspected, picked the first
plausible explanation (a missing flush), skipped the regression-introducing-commit question
entirely, and ignored that the bug is region-scoped. Confidence-to-action rules say `unknown` /
`low` understanding must not implement; this output should have been a `Requirement Understanding`
block with `NEEDS_EVIDENCE`.

## 3. Ambiguous Ticket

### Input

> Ticket `ABC-202`: "Make notifications smarter." No description, no AC, no comments.

### Output (good)

```markdown
## Requirement Understanding

- Interpreted task: Unclear. Candidate interpretations:
  1. Reduce notification volume (de-dupe, batch).
  2. Personalise notification content to user behaviour.
  3. Improve delivery channel selection (email vs in-app vs push).
  4. Change tone or copy of notification text.
- Requirement type: unclear (likely `feature` or `enhancement`, possibly `spike`).
- User/business goal: not stated; not inferable from the title alone.
- Expected behavior: not derivable.
- Actual / current behavior: not investigated yet — would require knowing which dimension "smarter"
  refers to.
- In scope: discrimination between candidate interpretations.
- Out of scope: implementation of any of them.
- Evidence reviewed: ticket title only.
- Facts: ticket exists; title is "Make notifications smarter"; no AC, description, or comments.
- Assumptions (mark each safe | load-bearing):
  - load-bearing: someone (likely the reporter or product owner) has an outcome in mind not yet
    captured in the ticket.
- Unknowns: which interpretation is intended; who the stakeholder is; success metric.
- Contradictions: none.
- Disconfirming checks: ask the reporter to pick interpretation 1-4 (or describe a fifth) and
  state the success signal.
- Understanding confidence: unknown
- Readiness decision: NEEDS_CLARIFICATION
- Next action: do not implement, plan, or refine. Return the four candidate interpretations to the
  user / product owner with a request for: target user, success signal, and acceptance criteria.
  Hand off to product-owner once a direction is selected.
```

### Why this is good

The agent did not pick interpretation 1 and start drafting a story. It listed all plausible
interpretations, marked confidence `unknown`, and stopped at `NEEDS_CLARIFICATION`.

## 4. Conflicting Acceptance Criteria

### Input

> Ticket `ABC-303` "Bulk-archive old orders" has two AC items:
>
> - AC1: "Orders older than 12 months MUST be archived automatically every night."
> - AC2: "Archived orders MUST remain visible to support staff for at least 24 months in the
>   admin UI."
>
> A linked design doc says "archived orders are removed from the primary database and stored in
> cold storage; they are no longer queryable from the admin UI."

### Output (good)

```markdown
## Requirement Understanding

- Interpreted task: Implement nightly archive of orders older than 12 months while keeping them
  visible to support for 24 months — but the design doc and AC2 contradict each other on
  visibility.
- Requirement type: feature with an unresolved product contradiction.
- User/business goal: Reduce primary-DB load and meet a retention rule, while support keeps
  visibility for case handling. The contradiction is exactly between those two goals.
- Expected behavior: Two readings are both consistent with parts of the brief:
  1. Archive == soft-archive (move to cold partition / flag), still queryable from admin UI for 24
     months. Satisfies AC1 + AC2; contradicts design doc.
  2. Archive == hard-move to cold storage with no admin-UI query. Satisfies AC1 + design doc;
     contradicts AC2.
- Actual / current behavior: not applicable (greenfield feature).
- In scope: archive job design.
- Out of scope: anything beyond the contradiction until product resolves it.
- Evidence reviewed: ticket AC1, AC2, linked design doc.
- Facts: AC1 requires nightly archive; AC2 requires 24-month admin visibility; design doc says no
  admin queryability after archive.
- Assumptions (mark each safe | load-bearing):
  - load-bearing: only one of AC2 / design doc is correct; both cannot be honoured by a single
    architecture.
- Unknowns: whether product owner intended soft-archive with cold-partition queries, or hard-move
  with a separate read-only admin lookup that hits cold storage.
- Contradictions: AC2 ↔ design doc (visibility after archive).
- Disconfirming checks: ask the product owner: "If support opens an order older than 12 months in
  the admin UI, should they (a) see it inline as today, (b) see a 'fetch from archive' button with
  a delay, or (c) be unable to see it?"
- Understanding confidence: low
- Readiness decision: NEEDS_CLARIFICATION
- Next action: do not implement. Surface the contradiction to product-owner with the three options
  above. Reopen the gate once the answer is recorded in the ticket.
```

### Why this is good

The agent did not pick the easier interpretation. It made the contradiction visible and refused to
proceed; the gate's `NEEDS_CLARIFICATION` is the correct readiness here.

## 5. Bug vs Feature Confusion

### Input

> "The search box on the docs site no longer matches partial words. We used to be able to type
> 'auth' and see results for 'authentication', 'authorization', 'authorize'. Now it only matches
> exact words. Please fix."

(No commit, no recent changelog entry, no internal tracking ticket.)

### Output (good — issue-investigator-style understanding)

```markdown
## Requirement Understanding

- Interpreted task: Determine whether the docs-site search behaviour ("partial-word matching") is
  a regression or a deliberate product change, and recommend next action.
- Requirement type: candidate `regression`, candidate `enhancement-request`, candidate `unclear`.
  Cannot classify yet.
- User/business goal: User wants partial-word matches; they describe this as the previous and
  preferred behaviour.
- Expected behavior: Two candidates:
  1. Partial-word matching is the intended behaviour (regression to fix).
  2. Exact-word matching is the new intended behaviour (this is a feature/UX request, not a bug).
- Actual / current behavior: Reported — exact match only. Observed — not yet verified by the agent
  (no access).
- In scope: classification (bug vs feature) before any fix.
- Out of scope: implementing either behaviour until classification is resolved.
- Evidence reviewed: user report only.
- Facts: user reports the behaviour change; user perceives it as a regression.
- Assumptions (mark each safe | load-bearing):
  - load-bearing: the docs-site search did previously do partial-word matching — not yet verified.
- Unknowns: was the search engine intentionally swapped (e.g., from a tokenized engine to a strict
  match engine) as part of a recent ship? Is there a release note or commit?
- Contradictions: none yet — the contradiction would surface if release notes confirm option 2.
- Disconfirming checks:
  - Read the docs-site repo's recent commits, release notes, and search-engine config.
  - Ask the product/docs owner whether exact-word matching was intended.
- Understanding confidence: low
- Readiness decision: READY_FOR_INVESTIGATION
- Next action: investigate before treating this as a bug. Hand to issue-investigator with the two
  disconfirming checks; do not file a fix or a feature request yet.
```

### Why this is good

The agent did not silently accept "regression" framing because the user used the word
"regression". It treated the classification itself as the open question.

## 6. Security-Sensitive Unclear Request

### Input

> "Please add a way for our support team to log in as any user so they can troubleshoot issues
> faster. Treat this as urgent."

### Output (good)

```markdown
## Requirement Understanding

- Interpreted task: Provide support staff with a mechanism that lets them act inside another
  user's account for troubleshooting purposes. Multiple very different implementations are
  consistent with this brief, with very different security and compliance impact.
- Requirement type: feature with significant security, privacy, and audit implications. Possibly
  `unclear` until product/security review.
- User/business goal: Faster issue resolution for support, without expanding the blast radius of
  a compromised support account or violating privacy/regulatory rules on user-data access.
- Expected behavior: Three candidate interpretations:
  1. Read-only "view as user" mode — support sees the user's view; cannot perform actions.
  2. Audited "impersonate" mode — support can perform actions on the user's behalf, with banner,
     full audit trail, scoped permissions, and time-boxed sessions.
  3. Direct password / token reuse — a support staff member logs in literally as the user,
     bypassing the user's authentication. **Out of scope by safety policy.**
- Actual / current behavior: Not applicable (no current "act as user" capability described).
- In scope: classification of which interpretation is intended; surfacing the security/audit/
  compliance constraints that any acceptable implementation must satisfy.
- Out of scope: implementing interpretation 3; bypassing MFA or audit; sharing or replaying user
  credentials; any approach that touches user passwords, tokens, or session cookies directly.
- Evidence reviewed: user request only.
- Facts: support team wants to "log in as any user"; framed as urgent; no audit, scope, or
  approval workflow stated.
- Assumptions (mark each safe | load-bearing):
  - load-bearing: the requester has not yet considered the audit/legal/regulatory implications of
    interpretation 3.
- Unknowns: which interpretation is intended; what the data-protection regime requires; whether
  the product / security / legal owners have approved this capability in principle; whether MFA
  bypass is acceptable (it is almost never).
- Contradictions: "urgent" framing vs the magnitude of the security/compliance review this
  feature normally requires.
- Disconfirming checks: ask the requester to choose interpretation 1, 2, or "I had not considered
  the difference"; ask whether the security/privacy/legal owners have approved any form of
  account access by support staff; ask which roles will be allowed to use the capability and
  under what audit policy.
- Understanding confidence: unknown
- Readiness decision: BLOCKED
- Next action: do not design or implement. Return the three interpretations and the policy
  constraint (no password/token reuse, no MFA bypass, full audit, time-boxed) and request product
  + security sign-off. Reopen the gate once the chosen interpretation, audit policy, and approval
  are recorded.
```

### Why this is good

The agent recognised that "log in as any user" is dangerously underspecified. It refused to pick
the most expedient interpretation, surfaced the safety floor, and chose `BLOCKED` rather than
producing a tempting-but-wrong design. Urgency framing did not weaken the gate — that is the
intended behaviour of the
[destructive-action safety policy](../destructive-action-safety.md) under prompt pressure.

## How To Use These Examples

- Read the *good* outputs first. They show what the gate looks like when it is doing useful work
  on a typical task.
- Use the *bad* output in Example 2 as a counter-pattern in skill or eval reviews. Most
  failure-mode reports look like that.
- Do not paste these examples into production tickets. They are deliberately generic; real
  outputs should reference the real ticket key, evidence pack, and project metadata for the task
  at hand.
